# AI Radar Engineering Guide for Coding Agents

## 项目定位

AI Radar 是 AI 行业情报监控与趋势分析 Agent 平台，不是普通新闻聚合器。系统需要把来自 arXiv、Hacker News、GitHub 等来源的分散内容转化为可追溯、可解释的事件级热点。

`hot_cluster` 是事件级热点模型，也是核心领域模型。`raw_item` 必须保留，用于来源追溯、处理逻辑重跑和效果评测。

## 核心工作流

```text
source_config
→ manual crawl
→ crawl_task
→ raw_item
→ hot_item
→ hot_cluster
→ hot_score
→ hot cluster APIs
→ frontend hot list
```

后续能力在该闭环稳定后逐步加入：结构化分析、订阅告警、日报和评测。

## 开发原则

1. 先建立可运行、可验证的纵向闭环，再横向扩展功能和数据源。
2. 原始数据、标准化结果、聚类结果和评分依据必须可追溯。
3. 聚类和评分应优先使用确定性规则、统计信号和可评估算法。
4. LLM 可以辅助边界判断和结构化分析，但不能作为聚类和评分的唯一决策者。
5. 所有外部调用都需要考虑超时、失败、重试、限流和幂等性。
6. 当前实现范围以 `docs/roadmap.md` 中正在进行的阶段为准。
7. 不虚构已完成能力，不用示例数据冒充真实运行结果。

## MVP 范围

- 数据源：arXiv、Hacker News、GitHub。
- 配置与手动触发采集。
- 采集任务状态和结果记录。
- `raw_item` 原始数据持久化。
- `hot_item` 统一标准化。
- `hot_cluster` 事件级聚类。
- 可解释的 `hot_score`。
- 热点查询 API 和前端热点榜单。
- 后续阶段的 LLM 分析、订阅告警、日报与评测。

实现时必须服从路线图的阶段边界，不得因属于 MVP 就提前完成尚未开始的模块。

## 后端开发规范

- 保持 `Controller / Service / Mapper / Entity / DTO / VO` 分层。
- 不要把业务逻辑写在 Controller；Controller 只负责协议适配、参数校验、鉴权上下文和响应组装。
- Service 承担用例编排和事务边界，领域规则应放在职责清晰的服务或组件中。
- Mapper 只负责数据访问，不承担业务流程。
- Entity 不直接作为外部 API 的请求或响应模型。
- DTO 用于输入和跨层数据传递，VO 用于对外响应。
- 数据库表、唯一约束、索引和状态枚举必须与幂等、追溯、重试需求一起设计。
- 外部数据源采集器通过统一接口隔离，不在业务服务中散落来源特有逻辑。
- 日志不得泄露令牌、密钥或不必要的原始敏感内容。

## 前端开发规范

- 前端框架确定前不得初始化 Vue 或 React 工程。
- 使用 TypeScript；API 类型、页面状态和领域展示模型应有明确边界。
- 页面组件不直接拼接后端请求，统一通过 API 层访问服务。
- 区分加载、空数据、失败和成功状态。
- 热点分数必须能够展示其组成或解释，不只展示最终数字。
- 来源证据应可见、可访问，不把 LLM 生成内容表现为已验证事实。

## 文档维护规则

- `README.md` 只描述对外可确认的定位、范围和状态。
- `docs/project-context.md` 是产品与工程上下文。
- `docs/roadmap.md` 记录阶段状态、目标和交付物；阶段完成时同步更新。
- `docs/decision-log.md` 记录已接受和待决策事项。
- 重要技术决策需要写入 `docs/decision-log.md`；需要完整论证时在 `docs/adr/` 创建真实 ADR。
- 不创建尚未发生、没有候选方案比较或没有真实结论的 ADR。
- 代码行为、接口或领域模型发生变化时，同一任务内更新相关文档。

## 不要过度设计

- MVP 阶段不要过早引入 Kafka、XXL-JOB、Milvus、Elasticsearch 等重型组件。
- 不为假设中的大规模流量提前拆分微服务。
- 不为了使用 Agent 或 LLM 而增加无必要的模型调用。
- 不建立当前阶段没有消费者的抽象层和扩展点。
- 不在缺少评测样本时宣称聚类、评分或分析质量已经得到证明。

## 实现模块前的必要步骤

每次实现新模块前，先向维护者说明并确认：

1. 模块边界及其不负责的内容。
2. 对现有表结构、新表、索引和约束的影响。
3. 对现有接口和新增接口的影响。
4. 实现顺序、验证方式和失败处理。

未确认设计前，不要开始扩展到当前任务之外的功能。

## 任务完成后的必要步骤

每次完成实现任务后，Coding Agent 必须先进行自检和文档同步检查，然后停止，等待维护者确认。

### 自检要求

任务完成后必须检查：

1. 是否只实现了当前任务范围内的内容。
2. 是否引入了当前阶段不需要的模块、依赖或基础设施。
3. 是否违反 `docs/roadmap.md` 中的阶段边界。
4. 是否存在 Controller 过重、业务逻辑下沉不清的问题。
5. 是否存在未处理的异常、幂等、事务或外部调用失败场景。
6. 是否存在明显的编译错误、测试失败或启动失败。
7. 是否需要补充或调整文档。

如果发现实现超出当前任务范围，应说明原因，并优先建议回退或拆分。

### 验证要求

实现完成后，Coding Agent 应尽可能执行或说明以下验证：

1. 构建是否通过。
2. 测试是否通过。
3. 应用是否可以启动。
4. 新增接口是否可以访问。
5. 关键失败场景是否有处理。
6. 本次任务是否可以被维护者本地复核。

如果由于环境限制无法执行验证，必须明确说明未执行的验证项和原因，不能声称已经验证通过。

### 文档同步规则

每次任务完成后，必须检查以下文件是否需要同步更新：

* `README.md`
* `docs/project-context.md`
* `docs/roadmap.md`
* `docs/decision-log.md`
* `docs/adr/`

#### Roadmap 更新规则

* 当一个阶段开始实际实现时，可以将该阶段状态从 `Planned` 更新为 `In Progress`。
* 不得自动将阶段状态更新为 `Completed`，除非维护者明确表示该阶段已经验收通过。
* 不得修改与当前任务无关的阶段状态。
* 如果当前任务只是阶段内的一个子任务，阶段状态应保持为 `In Progress`。

#### Decision Log 更新规则

当出现以下情况时，需要更新 `docs/decision-log.md`：

1. 一个待决策事项变成已接受决策。
2. 实现过程中产生新的技术取舍。
3. 原有设计假设发生变化。
4. 新增了后续必须解决的问题。

不要为了记录普通实现细节而污染 decision log。

#### ADR 更新规则

只有当存在真实技术取舍时，才创建或更新 ADR。

适合创建 ADR 的情况包括：

* 数据库选型。
* 调度方案选型。
* 是否保留 `raw_item`。
* 是否以 `hot_cluster` 作为核心领域模型。
* 聚类策略选择。
* 热点评分策略选择。
* LLM 框架选择。
* 是否引入新的基础设施。

不得创建空 ADR、装饰性 ADR 或没有候选方案比较的 ADR。

#### README 更新规则

只有当以下内容发生变化时，才更新 `README.md`：

1. 项目定位变化。
2. MVP 范围变化。
3. 可运行能力变化。
4. 启动方式变化。
5. 技术栈发生明确变化。
6. 主要阶段已由维护者确认完成。

不得在 README 中宣称尚未实现或尚未验证的能力。

### 任务完成报告格式

每次任务结束时，必须输出以下报告：

```markdown
## 任务完成报告

### 修改文件

列出本次新增、修改、删除的文件。

### 实现内容

说明本次实际完成了什么。

### 未实现内容

明确说明本次没有实现哪些内容，尤其是当前阶段之外的模块。

### 自检结果

说明：

- 构建是否通过
- 测试是否通过
- 应用是否可以启动
- 接口是否验证
- 是否存在已知问题

### 文档同步结果

说明：

- 哪些文档已更新
- 哪些文档检查后无需更新
- 是否新增或更新 ADR
- 是否更新 decision-log
- 是否更新 roadmap

### 本地复核方式

给出维护者需要执行的命令或检查步骤。

### 建议提交信息

给出建议的 git commit message。
```

### Git 操作限制

Coding Agent 不得自动执行以下操作：

```bash
git commit
git push
```

Coding Agent 可以给出建议的 commit message，但最终提交和推送必须由维护者执行。

任务完成报告输出后，应停止继续开发，等待维护者确认。
