# AI Radar Decision Log

本文件是技术与产品决策的入口。简短且已明确的决策可以直接记录在这里；需要完整背景、候选方案比较、后果和复审条件的决策，应在 `docs/adr/` 创建 ADR，并从本文件链接。

## Accepted Decisions

1. **AI Radar 不是普通新闻聚合器，而是事件级 AI 情报监控平台。**
   - 系统以 `hot_cluster` 作为核心业务对象，而不是只展示单条抓取内容。

2. **MVP 从 arXiv、Hacker News、GitHub 三个数据源开始。**
   - 三类来源分别代表论文研究、开发者讨论和开源项目动态。

3. **第一阶段先完成最小闭环，不急着做 LLM、订阅告警、日报和评测。**
   - 优先验证从数据源配置到热点榜单的真实纵向数据流。

## Pending Decisions

1. **PostgreSQL / MySQL / PostgreSQL + pgvector 如何选择。**
   - 需要结合结构化数据、JSON、向量聚类和部署复杂度进行比较。
   - 可能需要补充 ADR。

2. **任务调度用 Spring Scheduler / Quartz / XXL-JOB 如何选择。**
   - 需要结合 MVP 规模、持久化、重试、可观测性和部署成本进行比较。
   - 可能需要补充 ADR。

3. **是否保留 `raw_item` 原始数据。**
   - 需要明确追溯、重跑、存储成本和评测需求。
   - 该问题与项目原则直接相关，正式确认时应补充 ADR。

4. **`hot_item` 和 `hot_cluster` 的关系如何设计。**
   - 需要确定关系基数、聚类重算、主记录和事件合并策略。
   - 可能需要补充领域模型 ADR。

5. **去重聚类采用规则、embedding 还是 LLM。**
   - 需要比较准确率、成本、可解释性和边界样本处理。
   - 预计需要 ADR 和评测方案。

6. **热点评分采用规则模型还是 LLM 打分。**
   - 需要比较稳定性、可解释性、成本和可评测性。
   - 预计需要 ADR。

7. **LLM 框架用 Spring AI 还是 LangChain4j。**
   - 在进入结构化分析阶段前，根据模型支持、结构化输出和可观测性进行决策。

8. **前端使用 Vue 3 还是 React。**
   - 在初始化前端工程前，根据团队熟悉度、组件生态和展示需求进行决策。
